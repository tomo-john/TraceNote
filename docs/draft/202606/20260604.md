# Query Builder(クエリビルダ)

## 概要

データベースに対してSQLクエリを組み立てるためのツール。

## 基本

```php
<?php
use Illuminate\Support\Facades\DB;

$users = DB::table('users')
    ->where('active', 1)
    ->orderBy('created_at', 'desc')
    ->limit(10)
    ->get();
```

特徴:
- SQLに変換される前提で動く(`->get()`するまでクエリは実行されない=遅延評価)
- `where, join, groupBy, orderBy`などSQL操作に対応するメソッドが揃っている
- データベースの負荷を抑えられる(必要なデータだけ取得できる)

Eloquentモデルを使う場合も、内部的にはクエリビルダが使われる:

```php
<?php
$users = User::where('active', 1)->orderBy('created_at', 'desc')->get();
```

## Collection

クエリビルダの`->get()`などで取得した後のデータ(PHP配列的なもの)を操作するためのツール。

```php
<?php
$users = User::all(); // Collectionが返る

$activeUserNames = $users
    ->filter(fn ($user) => $user->active)
    ->map(fn ($user) => $user->name)
    ->sortBy('name');
```

特徴:
- メモリ上のデータに対して操作する(すでにDBから取得済み)
- `map, filter, reduce, groupBy, sortBy`など関数型っぽいメソッドが豊富
- 配列操作がチェーンで書けて読みやすい

## 大切な違い: いつ実行されるのか？

```php
<?php
// クエリビルダ:ここではまだSQLは実行されない
$query = User::where('active', 1);

// ->get() を呼んだ瞬間にSQLが実行され、結果がCollectionで返る
$users = $query->get(); // ここでSQL発行

// Collection:すでにPHP上のデータなので、ここからはDBアクセスなし
$names = $users->pluck('name');
```

## 使い分けの目安

| 状況                                   | 使うべきもの                          |
| -------------------------------------- | ------------------------------------- |
| 件数が多いデータから絞り込みたい       | クエリビルダの`where`など(DB側で絞る) |
| 取得済みデータを加工・整形したい       | Collection                            |
| 集計(SUM, COUNTなど)をDBにやらせたい   | クエリビルダ(`sum()`, `count()`)      |
| 複雑な配列操作(グループ化、ネスト処理) | Collection                            |

### よくないパターン:

```php
<?php
// 全件取得してからPHPで絞り込み -> DBに無駄な負荷
$activeUsers = User::all()->filter(fn ($u) => $u->active);
```

### 良いパターン:

```php
<?php
// DB側で絞り込んでから取得
$activeUsers = User::where('active', 1)->get();
```

## HasMany関係とクエリビルダの活用

HasManyなどのリレーションは、クエリビルダを返すメソッドとして定義されている。

=> そのままクエリビルダのメソッドが使える

### リレーションの基本構造

```php
<?php
class User extends Model
{
    public function posts()
    {
        return $this->hasMany(Post::class);
    }
}
```

この`posts()`メソッドは、`HasMany`というクエリビルダの一種のインスタンスを返している。

なので、`where`や`orderBy`などをそのままチェーンできる。

```php
<?php
// メソッドとして呼ぶ -> クエリビルダが返る -> さらに条件を追加できる
$user->posts()->where('published', true)->orderBy('created_at', 'desc')->get();
```

## プロパティとメソッドの違い

```php
<?php
// メソッドとして呼ぶ(括弧あり)
$user->posts()  // -> HasMany(クエリビルダ)が返る。追加で条件を付けられる

// プロパティとしてアクセス(括弧なし)
$user->posts    // -> Collectionが返る。もうDBには行かない
```

```php
<?php
// OK:まだクエリビルダなので条件を追加できる
$publishedPosts = $user->posts()->where('published', true)->get();

// エラー!Collectionに対してwhereは(SQL的には)使えない
$publishedPosts = $user->posts->where('published', true); // Collectionのwhereが呼ばれる(挙動が違う!)
```

Collectionに対しての`->where()`もエラーにならず動くが、これはCollectionの`whereメソッド`であり、SQLのWHEREではない！

=> 全件取得後にPHPでフィルタしているので、挙動が異なる

### 内部的な動き

`$user->posts`(プロパティアクセス)の場合、Laravelが裏で`__get()`マジックメソッドを使って:

```php
<?php
$user->posts(); // メソッドを呼ぶ
     ->getResults(); // クエリビルダのgetResultsを呼んで、結果をCollectionとしてキャッシュ
```

ということをやっている。

一度アクセスすると結果がキャッシュされるので、同じリクエスト中に`$user->posts`を何度呼んでもSQLは1回しか発行されない。

## N+1問題との関係

```php
<?php
// 悪い例: ユーザーごとにpostsを取得するSQLが発行される(N+1)
$users = User::all(); // 1回のSQL
foreach ($users as $user) {
    echo $user->posts->count(); // ユーザーの数だけSQLが発行される!
}

// Eager Loading(事前読み込み)で解決
$users = User::with('posts')->get(); // 2回のSQLで済む(usersとposts)
foreach ($users as $user) {
    echo $user->posts->count(); // すでに読み込み済みなのでSQL発行なし
}
```

### プチまとめ

| 書き方                | 戻り値                | 用途                                     |
| --------------------- | --------------------- | ---------------------------------------- |
| $user->posts()        | クエリビルダ(HasMany) | さらに条件を絞りたいとき                 |
| $user->posts          | Collection            | すでに絞り込み済みのデータを使いたいとき |
| $user->posts()->get() | Collection            | 明示的に条件付けてから取得               |
| User::with('posts')   | (Eager Loading)       | N+1回避、複数件まとめて事前取得          |
